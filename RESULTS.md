## Component 2 — Schema Extraction, Classification & IaC Generation

### Pattern Classification

Trained a scikit-learn `LogisticRegression` classifier to predict architecture pattern (monolith / microservices / event-driven) from graph-derived features (service count, route fan-out, DB dependency count, HTTP/queue external-call counts), replacing what would otherwise be a manually supplied label.

**Training data:** 600 synthetically generated feature vectors (200 per class), generated by `python/classifier/generate_training_data.py`. Not derived from real code — programmatically sampled to reflect each pattern's typical feature ranges, used only to teach the model each class's general shape.

**Held-out synthetic accuracy:** 1.000 (120-sample held-out split). Expected to be high — the synthetic generators are non-overlapping by construction (e.g. monolith is always exactly 1 service), so this mainly confirms the model can learn clean boundaries, not that it generalizes to real code.

**Real-code validation (the number that matters):** 3/3 (100%) — the trained classifier correctly labeled all 3 canonical sample apps, none of which were part of the synthetic training set.

| Sample | Expected | Predicted | Confidence |
|---|---|---|---|
| monolith | monolith | monolith | 0.992 |
| microservices | microservices | microservices | 0.869 |
| event-driven | event-driven | event-driven | 0.988 |

### IaC Generation & Validation

Generation is deterministic (Go `text/template`, no LLM, no network call at generation time). Validated end-to-end via `cmd/generate`, which chains extraction, classification, generation, and validation into a single command per target.

**Real results** (terraform v1.x, kubeconform v0.8.0, run against all 3 canonical sample apps):

| Pattern | terraform validate | kubeconform | Notes |
|---|---|---|---|
| monolith | PASS | PASS | 1 service, 5 routes, 1 RDS instance (postgres) |
| microservices | PASS | PASS | 3 services, 2 RDS instances (postgres + mysql), resolved cross-service HTTP calls |
| event-driven | PASS | PASS | 2 services, 1 shared SQS queue (deduplicated across publish/consume), worker service correctly has no exposed port |

**6/6 real validator checks pass** across all 3 canonical patterns — every piece of generated Terraform and Kubernetes output is syntactically and structurally valid according to the actual `terraform` and `kubeconform` CLIs, not a self-reported check.

### Known limitations (for viva / future work)
- Extraction rules are Gin-specific (`router.GET`, `sql.Open`, `os.Getenv` call patterns) — a different HTTP framework would need its own extraction rules added.
- Classifier trained on synthetic data due to limited real labeled examples (3 canonical samples); real-code validation is small-N (3/3) — a larger, more diverse real-code test set is future work.
- Generated IaC references a placeholder container image URI (`REPLACE_WITH_ECR_IMAGE_URI_FOR_<service>`) — building/pushing the actual image is out of scope, assumed to be a separate CI/CD step.
